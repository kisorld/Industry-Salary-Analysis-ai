# 计划书详细分析与实现范围评估

## 一、项目名称

行业薪酬洞察 AI 智能体

## 二、计划书目标概括

计划书提出的系统目标是：从公开招聘信息中获取岗位薪资数据，经过清洗、标准化、多维统计和可视化分析后，自动生成行业薪资对比报告，为企业招聘预算、调薪和薪酬竞争力评估提供参考。

核心功能包括：

1. 任务配置与筛选。
2. 招聘数据抓取或 CSV/Excel 导入。
3. 薪资字段清洗与标准化。
4. 按岗位、地区、经验、学历等维度进行统计。
5. 可视化看板展示。
6. AI 报告生成与 Word/PDF 导出。
7. 历史任务管理。

## 三、完整实现可行性判断

从 6 周项目周期看，计划书整体可实现，但需要分阶段交付，不能把所有模块都作为第2周基线目标。

可稳定实现的部分：

- CSV/Excel 导入。
- 薪资表达规则解析，例如 `10-15K`、`20K·14薪`、`1.5-2万`。
- 岗位、地区、经验、学历维度统计。
- 均值、中位数、P25、P75、P90 等指标计算。
- 基础可视化看板。
- Word 格式报告导出。
- 模板化或 API 驱动的报告文字生成。

需要后续增强或存在外部依赖的部分：

- BOSS 直聘、智联招聘等真实平台抓取。
- Playwright/Selenium 动态页面采集。
- Qwen2.5 API Tool Calling。
- PDF 导出。
- 历史任务数据库和多任务复查。
- 更完整的 BadCase 分析和模型评测。

## 四、主要风险

### 1. 真实招聘平台抓取风险

招聘网站页面结构变化快，且部分内容存在登录、验证码、动态加载或访问频率限制。若第2周把真实抓取作为主流程，演示失败概率较高。

处理建议：第2周保留 CSV/样例数据作为主路径，真实抓取作为后续增强和展示性能力。

### 2. AI API 稳定性风险

外部大模型依赖 API Key、网络、额度和响应格式。若没有严格 JSON Schema 和模板约束，报告可能引用不存在的数据。

处理建议：第2周采用模板化 AI 基线摘要，只引用程序计算出的统计值；第3周以后再接入 Qwen API。

### 3. PDF 导出环境风险

PDF 转换通常依赖 Office、LibreOffice、浏览器内核或字体环境，不同电脑结果可能不一致。

处理建议：第2周优先保证 Word 报告可生成；PDF 作为后续增强项。

### 4. 技术栈过重风险

Streamlit、FastAPI、Playwright、Qwen API、SQLite、PDF 导出全部并行实现，会增加前后端联调、环境配置和错误定位成本。

处理建议：第2周用轻量单体系统跑通端到端闭环。

## 五、第2周交付范围调整

第2周目标不追求最终系统完整度，而是证明核心业务链路可运行：

```text
CSV样例数据 -> 薪资解析 -> 数据清洗 -> 多维统计 -> 页面展示 -> Word报告导出
```

本次实现已覆盖：

- 内置 180 条基线岗位样例数据。
- 3 类岗位：数据分析、后端开发、产品经理。
- 4 个城市：北京、上海、广州、深圳。
- 3 类经验：1-3年、3-5年、5-10年。
- 薪资解析、异常标记、去重。
- 按岗位类别、城市、经验、学历统计。
- 浏览器演示页面。
- Word 报告导出。

## 六、技术栈调整结论

原技术栈适合作为最终目标：

- Streamlit 或 Vue。
- FastAPI。
- Requests/BeautifulSoup/Playwright。
- Qwen2.5 API。
- pandas/NumPy。
- SQLite。
- Plotly/ECharts。
- python-docx/PDF 转换。

第2周基线调整为：

- Python 标准库 Web 服务。
- CSV 数据导入。
- 规则化薪资清洗。
- Python 统计计算。
- HTML/CSS 图表。
- python-docx Word 报告。

这样做能保证第二周交付物可运行、可复核、可演示，同时不否定原计划书的最终方向。
