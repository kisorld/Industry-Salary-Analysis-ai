# 数据处理结果、成功失败案例与改进方案

## 1. 本轮处理结果概览

本轮使用项目内置基线数据集进行处理，数据规模为 180 条岗位记录。系统完成了 CSV 数据读取、pandas 批量清洗、薪资表达解析、岗位类别归一、多维统计和 AI 报告生成流程。

## 2. 成功案例

### 案例 1：标准 K/月 + 发薪月数

| 字段 | 内容 |
|---|---|
| 岗位 | 数据分析师 |
| 城市 | 北京 |
| 原始薪资 | 15-20K·13薪 |
| 解析结果 | 下限 15K/月，上限 20K/月，13 薪 |
| 月薪中值 | 17.5K |
| 年薪估算 | 195K-260K |

成功原因：薪资格式符合规则 `数字-数字K·N薪`，当前解析器可以稳定识别 K 单位、上下限和发薪月数。

### 案例 2：中文“万”单位

| 字段 | 内容 |
|---|---|
| 岗位 | 商业分析师 |
| 城市 | 北京 |
| 原始薪资 | 2.4-3.1万 |
| 解析结果 | 下限 24K/月，上限 31K/月，默认 12 薪 |
| 月薪中值 | 27.5K |

成功原因：系统已支持“万”到 K/月的单位换算。

### 案例 3：元/月格式

| 字段 | 内容 |
|---|---|
| 岗位 | 数据分析师 |
| 城市 | 上海 |
| 原始薪资 | 14000-21000/月 |
| 解析结果 | 下限 14K/月，上限 21K/月，默认 12 薪 |
| 月薪中值 | 17.5K |

成功原因：系统能够识别无单位但数值较大的薪资，并按元/月转换为 K/月。

### 案例 4：多维统计成功

系统能够按岗位类别、城市、经验、学历维度计算样本量、均值、中位数、P25、P75、P90、最低值和最高值。例如：

- 后端开发中位薪资最高，为 24.00K/月。
- 北京城市中位薪资最高，为 23.50K/月。
- 5-10 年经验样本中位薪资为 26.50K/月，高于 3-5 年和 1-3 年。

成功原因：统计计算由 pandas 完成，结果可复核，不依赖大模型直接生成数值。

## 4. 当前模型与系统能力缺口

### 4.1 薪资解析能力缺口

当前规则已覆盖：

- `10-15K`
- `10-15K·13薪`
- `1.5-2.2万`
- `18000-26000/月`
- `300-500元/天`
- `80-120元/时`
- `24-36万年薪`

仍需增强：

- `底薪10K+绩效`
- `15K起`
- `最高30K`
- `20-30K，另有年终奖`
- `10K-15K + 股票期权`
- `综合年薪30-50万`
- `税前/税后` 标注
- `兼职/实习/外包` 口径区分

缺口影响：复杂薪资结构如果直接参与统计，可能造成口径混乱；如果全部标记异常，则有效样本量下降。

### 4.2 岗位归一能力缺口

当前岗位类别主要依赖规则词典，例如：

- BI分析师、商业分析师 -> 数据分析
- Java后端开发工程师、服务端开发工程师 -> 后端开发
- 数据产品经理、B端产品经理 -> 产品经理

仍需增强：

- 新岗位名称自动归类能力不足。
- 复合岗位难以处理，例如“数据分析/产品经理”。
- 岗位级别未区分，例如初级、中级、高级、专家。
- 技能栈未参与分类，例如 Python、SQL、AIGC、推荐算法。

缺口影响：岗位类别口径不统一会直接影响岗位维度统计结果。

### 4.3 公开页面采集 PoC 缺口

当前采集 PoC 使用 HTML 文本提取和正则识别薪资表达，适合验证可行性，但不是稳定采集器。

主要缺口：

- 对动态加载页面支持不足。
- 对不同招聘网站 DOM 结构没有适配器。
- 无法处理登录、验证码、滚动加载等限制。
- 岗位标题、公司、城市和薪资之间的上下文绑定可能不稳定。
- 没有请求频率控制和采集日志。

缺口影响：真实平台采集结果可能缺字段、错配字段或采集不到数据。

### 4.4 Qwen Tool Calling 能力缺口

当前 AI 核心设计为：模型不直接计算薪资，而是调用工具读取程序统计结果，再生成报告。

已具备：

- `analyze_salary` 工具。
- `generate_docx_report` 工具。
- 缺少 Key 或网络失败时自动本地回退。
- Prompt 已限制不得编造数值。

仍需验证和增强：

- 当前运行环境中 Qwen 调用受网络权限影响，部分测试回退到本地模板报告。
- 需要确认所选模型是否稳定支持 Tool Calling。
- 需要记录每次 tool call 入参、出参和最终回答，便于评测。
- 需要增加“报告数值一致性校验”，防止模型输出与统计结果不一致。
- 需要增加结构化输出格式，例如 JSON Schema 或固定 Markdown 模板。

缺口影响：AI 报告的自然语言质量和稳定性取决于模型、网络、Prompt 和工具约束。

### 4.5 前端展示能力缺口

当前 Vue3 前端已支持样例分析、上传分析、采集 PoC、AI 报告和统计表展示。

仍需增强：

- 成功/失败样本未单独分栏展示。
- 异常原因没有图表化。
- 没有 BadCase 详情页。
- 没有历史任务列表。
- 没有模型调用日志查看。

缺口影响：答辩或验收时，系统对“为什么失败、如何改进”的解释能力还不够直观。

## 5. 改进方案

### 5.1 数据处理改进

| 优先级 | 改进项 | 说明 |
|---|---|---|
| P0 | 增加异常样本表 | 前端展示失败样本、失败原因、是否纳入统计 |
| P0 | 增加更多薪资格式规则 | 覆盖“起薪”“最高”“综合年薪”“底薪+绩效”等表达 |
| P0 | 增加统计口径说明 | 明确月薪、年薪、发薪月数、日薪/时薪转换规则 |
| P1 | 增加异常值检测 | 对极高/极低薪资进行 IQR 或分位阈值标记 |
| P1 | 增加人工修正入口 | 允许用户手动修正无法解析薪资 |

### 5.2 AI 核心改进

| 优先级 | 改进项 | 说明 |
|---|---|---|
| P0 | 固定模型输出结构 | 要求输出“样本说明、关键发现、建议、风险提示” |
| P0 | 增加数值一致性校验 | 检查模型回答中的数字是否来自统计结果 |
| P0 | 记录 Tool Calling 日志 | 保存工具调用参数、返回值和最终回答 |
| P1 | 引入 Few-shot 示例 | 给模型展示合格报告和不合格报告示例 |
| P1 | 将岗位/城市口径说明作为上下文 | 避免模型过度推断 |

### 5.3 采集 PoC 改进

| 优先级 | 改进项 | 说明 |
|---|---|---|
| P0 | 明确采集边界 | 只采集公开页面，不绕过登录、验证码和付费墙 |
| P0 | 增加采集日志 | 保存 URL、时间、成功条数、失败原因 |
| P1 | 为不同网站设计解析适配器 | 每个平台单独维护字段提取逻辑 |
| P1 | 增加请求频率限制 | 避免高频访问 |
| P2 | 引入 Playwright | 仅用于公开动态页面，不绕过访问限制 |

### 5.4 前端改进

| 优先级 | 改进项 | 说明 |
|---|---|---|
| P0 | 增加成功/失败样本 Tab | 分开展示清洗成功和 BadCase |
| P0 | 增加异常原因统计图 | 展示面议、无法解析、上下限异常等问题分布 |
| P1 | 增加 AI 调用状态 | 展示 qwen_tool_calling 或 local_fallback |
| P1 | 增加报告预览 | 下载 Word 前先查看报告正文 |
| P2 | 增加历史任务 | 支持回看每次分析结果 |

## 6. 下一轮迭代目标

建议下一轮优先完成以下闭环：

1. 前端新增 BadCase 面板，展示失败样本和失败原因。
2. 后端新增 `/api/badcases` 接口，返回异常数据列表和异常原因统计。
3. 薪资解析新增 8-10 类复杂表达测试。
4. Qwen 输出增加 JSON Schema 或固定字段，降低报告格式波动。
5. 增加模型输出数值一致性检查，确保报告中的数值来自 pandas 统计结果。
6. 采集 PoC 增加采集日志和字段错配提示。

## 7. 当前结论

当前系统已经具备可演示的基线能力：数据导入、批量清洗、薪资解析、多维统计、AI 报告生成和前后端展示均已打通。主要能力缺口集中在复杂薪资表达、真实页面稳定采集、AI 输出一致性校验和 BadCase 可视化展示。

因此，后续优化重点不应是盲目增加大模型能力，而应围绕“规则可复核 + AI 受约束解释 + BadCase 可追踪”推进。
