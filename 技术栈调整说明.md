# 技术栈调整说明

## 是否能按原计划完整实现

原计划书覆盖了数据采集、清洗、统计、可视化、AI 报告、Doc/PDF 导出、历史任务管理等完整链路。这个方向合理，可以作为 6 周最终目标推进。

但第2周要求交付《基线评测报告》和可运行演示系统。如果此阶段坚持同时实现真实平台抓取、Qwen API Tool Calling、FastAPI 服务化、PDF 导出和历史任务管理，风险会集中爆发：

- 招聘平台页面可能动态加载，且可能需要登录、验证码或频率控制。
- 外部大模型需要 API Key、网络和响应格式约束，演示稳定性不可控。
- PDF 导出依赖系统字体、Office/LibreOffice 或浏览器渲染环境。
- 前后端拆分和数据库会增加联调成本，不利于第二周验收。

## 第2周基线技术栈

| 层面 | 调整后技术 | 原因 |
|---|---|---|
| 界面 | Python 标准库 http.server | 不依赖额外安装，保证可运行 |
| 数据来源 | CSV 样例数据 + 上传 CSV | 避开真实抓取不稳定，满足基线闭环 |
| 清洗 | 规则解析薪资表达 | 结果可复核，适合评测准确率 |
| 分析 | Python 标准库统计函数 | 计算均值、中位数和分位值 |
| 可视化 | HTML/CSS 柱状图 + 表格 | 不依赖 Plotly/Matplotlib |
| AI 报告 | 模板化智能摘要 | 严格引用统计结果，避免幻觉 |
| 导出 | python-docx | 当前环境已具备，优先保障 Word 交付 |

## 后续增强路线

第3周：加入 pandas 批量处理、更多薪资格式测试、真实公开页面采集 PoC。

第4周：引入 Streamlit 或 Vue/FastAPI，完善测试报告和异常数据处理。

第5周：接入 Qwen2.5 API，采用 JSON Schema 和固定 Prompt 约束输出。

第6周：加入 SQLite 历史任务、PDF 导出、演示视频和最终答辩材料。
